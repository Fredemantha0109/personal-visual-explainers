# 知識ストック図解スキル（公認会計士・古屋さん専用）

## このスキルの目的

**「見返したとき、1分で核心が思い出せる」** 図解を作る。

- 読者は自分ひとり。他者への説明ではなく、**自分の記憶を呼び起こすトリガー**として使う
- 一度学んだことを何度も検索し直さないよう、核心・構造・実務場面をセットでストックする
- 対象ドメイン: AI/テック用語 ／ 会計・税務・監査 ／ 会社法・金融商品取引法 ／ 上場準備・PMI・内部統制・組織再編

---

## 依存ファイル

- `references/html-structure.md` — HTMLテンプレートとTailwindクラス一覧
- `references/content-guide.md` — セクション設計・情報の絞り方
- `references/exemplar.md` — 模範解答パターン（コピペ可能なHTMLブロック）

## ワークフロー

### Step 0: トピックを把握する

ユーザーが指定したトピックを確認し、以下を判断する:

| 判断項目 | 考え方 |
|---------|--------|
| **ドメイン** | AI/テック ／ 会計・税務・監査 ／ 法律・規制 ／ 経営・マネジメント |
| **トピックの種類** | 用語の定義 ／ 仕組み・プロセス ／ 比較・違い ／ 最新動向 |
| **なぜ何度も確認してしまうか** | 混乱しやすいポイントはどこか |

### Step 1: ウェブで情報収集する（2〜3回）

**まず `references/content-guide.md` の「推奨情報源」テーブルを確認し、ドメインに対応した公式ソースを優先的に参照する。** 一般的なウェブ検索より先に公式ソースに当たること。

正確な定義・最新情報・実務への接点を調べる。以下の観点でクエリを組み立てる:

1. **正確な定義**: `content-guide.md` の推奨情報源テーブルに従い、ドメイン対応の公式ソースから原文を取得
2. **実務への接点**: 上場準備・監査・内部統制・PMI等での具体的な登場場面
3. **混乱ポイント**: 類似概念との違い、誤解されやすい点

収集した情報は Step 2 の6セクション（核心ひとこと・全体構造・押さえる3点・実務場面・混乱ポイント・関連キーワード）の観点で整理する。出典URLも記録しておく。

### Step 2: コンテンツを設計する

`references/content-guide.md` を参照して、以下6つのセクションを設計する:

| セクション | 内容 |
|-----------|------|
| **① 核心ひとこと** | 1〜2文の本質的な定義 |
| **② 全体構造図** | フロー／階層／比較のビジュアル |
| **③ 押さえる3点** | 最重要ポイントを3つに絞る |
| **④ 実務場面** | 古屋さんの業務のどこで登場するか |
| **⑤ 混乱しやすいポイント** | 類似概念との違い・誤解しがちな点 |
| **⑥ 関連キーワード** | 関連用語を小さくリスト |

各セクションの設計指針・ドメイン別の書き方は `references/content-guide.md` 参照。

### Step 3: HTMLを生成する

`references/html-structure.md` と `references/exemplar.md` を参照してHTMLを生成する。禁止事項は「守ること（禁止事項）」セクション参照。

**⑥ 関連キーワードのリンク付与**:

⑥関連キーワードを生成する前に、`output/` ディレクトリ内の既存HTMLファイル（`index.html` 除く）の `<!-- DIAGRAM-META` ブロックを読み取り、各ファイルの `title`・`tags`・ファイル名（スラッグ）・`domain` を把握する。

各キーワードについて:
1. 既存図解のいずれかの `title` またはカンマ区切りの `tags` と完全一致 or 部分一致するか照合する
2. **一致した** → リンク付きバッジ（`<a href="./スラッグ.html">`）で出力。カラーは図解先ドメインの色を使う
3. **一致しない** → 通常の `<span>` バッジで出力

HTMLパターンは `references/html-structure.md` の「⑦ 関連キーワード」セクション参照。

**鮮度インジケーター（Freshness Badge）の挿入**:

ページのヘッダーブロック内、ドメインバッジの隣に鮮度バッジ要素を追加する。また、`</body>` 直前の `lucide.createIcons()` スクリプト行の **直前**に、鮮度バッジ計算スクリプトを挿入する。

HTMLパターンと JS スニペットは `references/html-structure.md` の「鮮度インジケーター」セクション参照。

### Step 3.5: 品質セルフチェック

HTMLを生成したら、**保存前**に以下のチェックリストを自己採点する。NGが1つでもあれば即座に修正してから Step 4 へ進む。

| # | チェック項目 | 基準 | NGの場合の対処 |
|---|------------|------|-------------|
| 1 | **核心ひとこと** | 1〜2文以内か。「この概念がなぜ存在するか（Why）」まで含んでいるか | 文言を削る・Why視点を1文加える |
| 2 | **実務場面** | 古屋さんの業務文脈が2〜4個あるか | 1個しかない場合は具体的な場面を追加 |
| 3 | **混乱ポイント** | 1個以上あるか。「× よくある誤解 / ○ 正しい理解」の対比形式になっているか | 形式が崩れていれば対比形式に修正 |
| 4 | **全体構造図** | A〜F のいずれかのパターンで構造化されているか（箇条書きになっていないか） | 箇条書きになっていれば図式化 |
| 5 | **情報量** | 「教科書になっていないか」。専門家が2分で読み終わる量か | 冗長な背景説明・当然の前提知識を削る |
| 6 | **ドメインカラー** | ドメインバー・バッジ・番号バブル・タイトル文の色が一貫しているか | ドメインカラー表に照らして修正 |

全て✓になったら Step 4 へ進む。

### Step 4: ファイルを保存する

1. `output/` ディレクトリがなければ作成する
2. スラッグを決める（例: `内部統制` → `naibutosei`, `IFRS` → `ifrs`）
3. `output/{スラッグ}.html` に保存する
4. HTMLファイルの **`<body>` タグの直後**にメタデータコメントを挿入する。フォーマットは `references/html-structure.md` の「メタデータコメント」セクション参照。

### Step 4.5: インデックスページを更新する

図解の保存後、**必ず** `output/index.html` を再生成する。

#### 生成手順

1. `output/` ディレクトリ内の `.html` ファイルを一覧取得する（`index.html` 自身は除外）
2. 各ファイルから `<!-- DIAGRAM-META` ブロックを読み取り、メタデータを抽出する
3. `reviews` の値が 1 以上のファイルは「再確認あり」として記録する
4. ドメイン別にグループ化し、各グループ内は `created` の降順（新しい順）で並べる
5. `output/index.html` を生成して保存する

#### index.html の構成

```
出力: output/index.html

ページ構成:
├── ヘッダー: 「知識ストック」タイトル + 総件数 + 最終更新日
├── ドメイン別セクション（存在するドメインのみ表示）
│   ├── AI・テック        → 青系カラー
│   ├── 会計・税務・監査  → 緑系カラー
│   ├── 法律・規制        → 紫系カラー
│   └── 経営・マネジメント         → 琥珀系カラー
└── 各カード: タイトル / 作成日 / 再確認バッジ / タグ
```

HTMLテンプレートは `references/exemplar.md` の「インデックスページテンプレート」セクション参照。

#### 「また調べた」ワークフローとの連携

「また調べた」で `reviews` カウントが更新されたあとも、同じ手順でインデックスを再生成する。

### Step 4.8: surge.sh にデプロイする

インデックス更新後、**必ず**図解をデプロイする。

#### 事前確認

```bash
node --version
```

バージョン番号が表示された → 「デプロイ実行」へ進む。  
`command not found` → 「Node.js がインストールされていません。`node.js.org` からインストール後、もう一度お試しください。」と伝えて終了。

#### デプロイ実行

```bash
bash .claude/skills/creating-visual-explainers/scripts/deploy-diagram.sh output/{スラッグ}.html cpa-{スラッグ}
```

- デプロイ先ドメイン: `diagram-cpa-{スラッグ}.surge.sh`
- スクリプトが自動でURLをクリップボードにコピーし、ブラウザで開く
- `deploy-history.log` にデプロイ履歴が記録される

#### 初回の場合（Surge 未登録）

ターミナルにメールアドレス・パスワードの入力を求められたら以下を伝える:

> 初回のみアカウント登録が必要です。メールアドレスを入力して Enter → パスワードを決めて Enter。次回以降は不要です。

#### エラーが出た場合

| エラー | 対処 |
|-------|------|
| `surge: not found` | `npm install -g surge` を実行してから再試行 |
| `Login required` | `npx surge login` を実行 |
| その他 | 何が問題で何をすれば解決するかを平易に説明する |

### Step 5: 完了を報告する

#### デプロイ成功時

https://diagram-cpa-{スラッグ}.surge.sh

## 「また調べた」ワークフロー

「また〇〇を調べてしまった」「〇〇を再確認」と言われたとき、**新しく作るのではなく既存の図解を育てる**。

### トリガーとなる表現

- 「また〇〇を調べてしまった」
- 「〇〇を再確認したい」
- 「〇〇の図解に再確認を記録して」

### ワークフロー

#### Step A: 既存ファイルを探す

`output/` ディレクトリ内のHTMLファイルを確認し、該当トピックのファイルを特定する。

- 見つかった → Step B へ進む
- 見つからない → 「この図解はまだ作成されていません。新しく作りますか？」と聞く

#### Step B: 何が思い出せなかったかを聞く

```
「【トピック名】の図解が見つかりました。
今回、どこが思い出せなかったですか？（省略可）」
```

ユーザーの回答を「記憶の弱点メモ」として記録する。省略された場合は空欄のまま記録する。

#### Step B.5: 弱点パターンを分析する（reviews ≥ 2 の場合）

`reviews` が **2 以上** のとき、既存の `<!-- 再確認履歴 -->` ブロックの全エントリを読み取り、以下を実施する。

**1. 繰り返しパターンの検出**

過去の「思い出せなかった箇所」を照合し、同じセクションや同じテーマが複数回登場していないか確認する。

| 検出パターン | AIの対応 |
|------------|--------|
| 同じセクション（例: 実務場面）が2回以上 | そのセクションの強化を能動的に提案する |
| 「ぼんやりしていた」「核心がわからなかった」が2回以上 | 核心ひとことの文言見直しを提案する |
| 3回以上同じ箇所 | 「このセクションを根本から書き直すことを推奨します」と提案 |
| 過去の記録が全て「記録なし」 | パターン分析は行わず Step C に進む |

**2. 今回の回答が省略された場合**

過去の履歴から「弱そうなセクション」を推測し、以下のように提案する:

```
過去の再確認履歴を確認しました（通算 N 回目）。
過去の記録: 【過去の思い出せなかった内容を1〜2行で要約】

繰り返し【〇〇セクション】が弱点になっている可能性があります。
今回もそこを強化しますか？（「はい」「いいえ」または具体的な箇所を教えてください）
```

**3. reviews ≥ 3 かつ同じ箇所が繰り返されている場合**

Step D での強化に加え、以下を能動的に提案する:
- 核心ひとことの文言を根本から書き直す
- 混乱ポイントに「なぜ毎回ここを忘れるか」の注記を追加する（例: 「⚑ 3回この点で詰まった」ラベル）

#### Step C: 再確認履歴とメタデータを更新する

既存HTMLファイルの **ヘッダー直後**（`<!-- 再確認履歴 -->` コメントの位置）を探し、再確認バッジブロックを更新する。

**初回の再確認（バッジブロックがまだない場合）**: バッジブロックを新規挿入する。  
**2回目以降（バッジブロックが既にある場合）**: 履歴リストに1行追加し、回数カウントを更新する。

バッジブロックの仕様は `references/html-structure.md` の「再確認履歴バッジ」セクション参照。

あわせて、ファイル内の `<!-- DIAGRAM-META` ブロックを以下の2フィールド更新する:
- `reviews:` → 現在の再確認回数（整数）
- `last-reviewed:` → 本日の日付（YYYY-MM-DD）

また、鮮度バッジ要素の `data-last-reviewed` 属性も本日の日付に更新する。これにより、次にページを開いたとき鮮度インジケーターが「最終確認からの経過日数」を正しく表示する。

#### Step D: 弱点部分を強化する

Step B のユーザー回答 **および** Step B.5 の分析結果を組み合わせて、該当セクションのHTML記述を強化する。ユーザーが省略した場合でも、B.5 の分析で特定した弱点があれば能動的に強化を行う。

| 思い出せなかった内容 | 強化方法 |
|---------------------|--------|
| 「核心ひとことがぼんやりしていた」 | 核心ひとことの文言を見直す |
| 「〇〇と〇〇の違いが混乱した」 | 混乱しやすいポイントに比較行を追加 |
| 「実務のどこで使うか忘れた」 | 実務場面に具体例を1行追加 |
| 「全体像がつかめていなかった」 | 構造図をより明快に書き直す |
| reviews ≥ 3 かつ同じ箇所が3回以上 | 核心ひとことを根本から書き直し、混乱ポイントに繰り返し弱点の注記を追加 |

#### Step E: インデックスを再生成・デプロイして完了を報告する

Step 4.5 と同じ手順で `output/index.html` を再生成する（`reviews` カウントが変わったため）。

次に Step 4.8 と同じ手順で更新した図解を再デプロイする（既存ドメインに上書きされる）。

その後、完了を報告する:

```
再確認を記録しました: 【図解のタイトル】

記録内容:
- 日時: YYYY-MM-DD（通算 N 回目）
- 思い出せなかった箇所: 【記録した内容 or「記録なし」】
- 図解の強化: 【行った変更 or「なし」】

【reviews ≥ 2 のときのみ追記】
弱点パターン分析:
- 【繰り返し弱点のセクション名】が N 回登場しています
- 【今後の改善提案 or「現時点では特定のパターンは見られません」】
```

## 守ること（禁止事項）

- **`<style>` タグを追加しない** — スタイリングはTailwind CSSクラスで行う
- **`<script>` タグを追加しない** — 鮮度インジケーター専用の日付計算スクリプト（`references/html-structure.md` 所定のもの）は例外として許可。それ以外のJavaScript禁止
- **絵文字を使わない** — アイコンはLucide iconsを使う
- **インタラクティブ要素を入れない** — トグル・アニメーション・クリック開閉禁止
- **外部リソースを追加しない** — CDN以外の外部読み込み禁止
- **説明過多にしない** — 「1分で思い出せる」が基準。教科書ではなく、記憶トリガー
- **読者レベルに言及しない** — 専門家向けなので専門用語はそのまま使ってよい

## 品質基準

**「3ヶ月後の自分が見返したとき、1分で核心を思い出せるか」**

- 核心ひとことを読んで「そうそう、そういうやつ」となるか
- 全体構造図を見て位置づけが即わかるか
- 実務場面を読んで「この仕事のこの文脈だ」と紐付けられるか
- 混乱ポイントを読んで「この類似概念との違いか」と確認できるか
